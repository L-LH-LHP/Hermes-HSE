# 文档加密存储和解密功能使用指南

## 功能概述

已实现的文档加密存储和客户端解密功能，允许：

1. ✅ **服务器端加密存储**：原始文档被加密后存储在服务器
2. ✅ **客户端解密查看**：客户端可以下载加密文档并解密查看
3. ✅ **服务器无法查看明文**：服务器只存储加密内容，无法解密

## 使用流程

### 步骤1：准备原始文档

原始文档应该放在适当的位置。如果是使用Enron邮件数据集，文档已在`maildir/`目录中。

### 步骤2：加密存储文档（服务器初始化）

在服务器启动时，需要将原始文档加密存储。这需要在`server.cpp`的`init()`函数中添加文档加密逻辑。

**注意**：当前版本的`document_storage.cpp`提供了加密存储功能，但需要在服务器初始化时调用。

### 步骤3：搜索文档

用户通过Web界面搜索关键词，系统返回匹配的文件ID列表。

### 步骤4：查看解密文档

1. 在搜索结果中点击文件ID
2. 系统会自动：
   - 从服务器获取加密文档
   - 在客户端使用密钥解密
   - 在浏览器中显示解密后的内容

## API接口说明

### 获取并解密文档

**请求**:
```bash
POST /api/document
Content-Type: application/json

{
    "writer_id": 0,
    "file_id": 1
}
```

**响应**:
```json
{
    "success": true,
    "writer_id": 0,
    "file_id": 1,
    "content": "解密后的文档内容...",
    "encoding": "utf-8",
    "size": 1234
}
```

## 安全性保证

### 服务器端

- ✅ 只存储加密文档（`.enc`文件）
- ✅ 使用AES-256-CTR加密
- ✅ 密钥派生基于文件ID和写入者ID
- ✅ 服务器无法访问解密密钥

### 客户端解密

- ✅ 解密密钥在客户端派生
- ✅ 解密过程在客户端完成
- ✅ 服务器只返回加密文档，无法看到明文

## 注意事项

1. **文档初始化**：在首次使用前，需要将原始文档加密存储
2. **密钥管理**：当前使用简化版密钥派生，生产环境需要集成HICKAE系统的读者密钥
3. **存储路径**：确保`encrypted_docs/`目录有写入权限
4. **性能考虑**：大文件解密可能较慢，建议添加进度提示

## 下一步集成

要将此功能完全集成到系统中，需要：

1. **修改服务器初始化**：在`server/server.cpp`的`init()`函数中，为每个文件调用`DocumentStorage::encrypt_and_store()`
2. **改进密钥管理**：使用HICKAE系统的读者密钥进行密钥派生
3. **添加访问控制**：确保只有授权的读者可以解密文档

## 测试建议

1. 先测试小文件（< 1KB）
2. 测试文本文件和二进制文件
3. 验证服务器确实无法查看明文
4. 测试多个写入者的文档
